## `/kanban-explore <topic>` — Explore & Plan

**When to use**: You have a vague idea or problem but don't know *how* to implement it.
This skill explores first, reports direction, then seeds the kanban board with phased tasks.
**This skill does NOT write code.**

---

### Procedure

```
① Clarify the topic (if too vague)
   If the user's topic is fewer than 10 words or lacks context, ask one focused question:
   - What problem are you trying to solve?
   - Is there a specific area of the codebase you suspect is involved?
   Keep it brief — one round only.

② Deep codebase exploration (Task → Explore agent)
   Launch an Explore subagent with thoroughness="very thorough".

   The Explore agent MUST investigate:

   A. ENTRY POINTS & STRUCTURE
      - Top-level directories and their roles
      - Main entry files (main.ts, index.ts, app.ts, etc.)
      - Config files (vite.config, tsconfig, package.json)

   B. RELEVANT CODE
      - Files, modules, components related to the topic
      - Existing patterns for similar features
      - Data flow: how data moves through the system

   C. PAIN POINTS & GAPS
      - Missing abstractions or obvious duplication
      - Areas where the topic touches multiple modules
      - Potential conflicts or dependencies

   D. TECHNOLOGY CONSTRAINTS
      - Libraries already in use (check package.json)
      - Framework patterns (what's idiomatic here?)
      - Build/test tooling

③ Write the Exploration Report
   Format:

   ---
   ## Exploration Report: <topic>
   *Explored: <timestamp> | Codebase: <project>*

   ### Current State
   [2–4 sentences: what exists today relevant to this topic]

   ### Key Findings
   - Finding 1 (with file path reference)
   - Finding 2 (with file path reference)
   - ...

   ### Possible Directions
   Present 2–3 concrete implementation directions, each with:

   #### Direction A: <name>
   **Approach**: [1–2 sentences]
   **Pros**: ...
   **Cons**: ...
   **Estimated complexity**: Low / Medium / High
   **Files likely touched**: [list]

   #### Direction B: <name>
   ...

   ### Recommended Direction
   [Which direction you recommend and why, based on the codebase evidence]
   ---

④ Present report + ask user to choose direction
   Use AskUserQuestion with:
   - One option per direction found
   - "Cancel (don't create tasks)" option

⑤ Generate phased kanban tasks
   Based on the chosen direction, create 3–7 tasks in logical implementation order.
   Each task:
   - title: concise imperative ("Add X", "Refactor Y", "Integrate Z")
   - description: uses the refined template from kanban-refine (Goal / Scope / Requirements / Acceptance Criteria)
   - priority: inferred from position (first tasks = high, later = medium/low)
   - level: L2 or L3 based on complexity of each step
   - tags: relevant module/area tags
   - description MUST include this block at the bottom:

     ---
     ## Exploration Context
     *Auto-generated by /kanban-explore on <timestamp>*
     **Direction chosen**: <Direction name>
     **Report summary**: <2–3 sentence summary of why this direction was chosen>
     **Prerequisite tasks**: <list sibling task IDs if applicable, else "none">

   Create tasks via API:
   ```bash
   curl -s -X POST http://localhost:5173/api/task \
     -H 'Content-Type: application/json' \
     -d "{
       \"title\": \"...\",
       \"project\": \"$PROJECT\",
       \"priority\": \"high\",
       \"level\": 3,
       \"description\": \"...\",
       \"tags\": \"...\"
     }"
   ```

   Fallback (no dev server):
   ```bash
   sqlite3 ~/.claude/kanban-dbs/$PROJECT.db \
     "INSERT INTO tasks (project, title, description, status, priority, level, tags, created_at)
      VALUES ('$PROJECT', '...', '...', 'todo', 'high', 3, '...', datetime('now'))"
   ```

⑥ Output final summary
   Print a table of created tasks:

   | # | ID  | Title                  | Priority | Level |
   |---|-----|------------------------|----------|-------|
   | 1 | 42  | Add X                  | high     | L3    |
   | 2 | 43  | Refactor Y             | medium   | L2    |
   ...

   Then print:
   > ✅ Exploration complete. N tasks created in `todo` for project `<PROJECT>`.
   > Run `/kanban-refine <ID>` on any task to add more detail before starting.
   > Run `/kanban-run` when ready to execute.
```

---

### Guardrails

- **No implementation**: This skill must NOT write, edit, or create source files.
- **No assumptions**: If the codebase has no clear pattern for something, say so explicitly.
- **Evidence-based**: Every claim in the report must cite a file path or code pattern found.
- **Honest about uncertainty**: If there is only one sensible direction, say so instead of fabricating alternatives.
- **Task granularity**: Each task should be completable independently in one pipeline run. Split tasks that touch more than 3 unrelated files.
